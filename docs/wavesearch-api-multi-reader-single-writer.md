# wavesearch-api: multi-reader / single-writer architecture

Modeled directly on **picowal's own replication engine**
(`picoweb/src/picowal_repl.{c,h}`, `picowal_repl_client.{c,h}`,
`picowal_gossip.{c,h}`) -- the same primary/replica/gossip protocol,
ported from "replicate a raw WAL byte-range over HTTP" to "replicate a
JSON blob manifest over HTTP," because wavesearch-api's durable state
*is* a set of named JSON blobs (`CATALOG_PRODUCTS_BLOB`,
`VECTOR_CACHE_BLOB`, etc. -- see `app.py`'s blob-path constants) rather
than picowal's single append-only sector-addressed log.

## The problem

`wavesearch-api` runs as 3 replicas today (see
`k8s/base/wavesearch-api-deployment.yaml`), but:

1. **Each pod's in-memory search runtime (BM25 postings, the numpy
   vector matrix, promotions index, autocomplete vocabulary) is
   process-local**, rebuilt only in the pod that happens to handle a
   given `POST /search/ingest/*` / `POST /search/admin/*` request. The
   other pods never see that mutation -- three pods behind one LB can
   silently return **different search results for the same query**
   depending on which pod an LB round-robins a request to.
2. Without a shared Azure Blob Storage configuration (the current k8s
   manifest sets none), the underlying **persisted JSON blobs
   themselves aren't shared either** (`LocalBlobStore` writes to each
   pod's own `emptyDir`/tempdir) -- so even a pod restart doesn't
   recover state consistently across the fleet.

This is exactly the shape of problem picowal's replication engine
solves for the C `pico*` stack: **one elected writer owns mutations
and a durable log; replicas pull that log over HTTP and replay it
locally; no shared network filesystem is required.** The same pattern
maps cleanly onto wavesearch-api's JSON-blob state.

## What gets replicated (and what deliberately doesn't)

Mirroring `app.py`'s existing blob-path constants, the **tracked
blobs** (mutated only through admin/ingest endpoints, and required for
search-result consistency across pods) are:

- `CATALOG_PRODUCTS_BLOB`, `ENRICHMENT_CACHE_BLOB`, `VECTOR_CACHE_BLOB`
- `RULES_BLOB`, `REDIRECTS_BLOB`, `PROMOTIONS_BLOB`
- `RANKING_OBJECTIVE_BLOB`, `ML_RANKER_BLOB`

**Deliberately excluded** (same "single-writer for correctness-critical
state, AP/per-pod for everything else" split the codebase's own
comments already assume): `VISITOR_SIGNALS_BLOB` and
`PRODUCT_PERFORMANCE_BLOB` -- per-visitor personalization signals and
product performance counters are high-write-volume, generated by
*every* search/event request, and already documented in `app.py` as
eventually-consistent, per-pod telemetry ("the model is already
updated instantly either way, this only bounds how much blob-write
work happens"). Routing every click/view event through a single writer
would defeat the entire point of horizontal read scaling for zero
correctness benefit -- these stay exactly as they are today.

## Protocol (mirrors picowal_repl's status/stream contract)

**Data plane** (served only by the current writer, gated by
`X-PW-Write-Token` exactly like picowal's `api_require_write_token`):

```
GET /repl/status
  -> 503 if this node is not currently the writer (mirrors
     picowal_repl_enabled()'s gate)
  -> 200 {"role":"writer","node_id":"...","generation":N,
          "blobs":{"cache/catalog-products.json":"sha256:...", ...}}

GET /repl/blob?path=<one of the tracked blob paths>
  -> 200 text/plain, the blob's current raw content
  -> 404 if path isn't one of the tracked blobs (allowlist, not
     arbitrary path passthrough)
```

**Control plane** (gossip leader election, always available, also
token-gated):

```
GET  /gossip/status  -> {"self":,"term":,"candidate":,"votes":,
                          "followers":,"quorum":,"promoted":,"known_leader":}
POST /gossip/vote     -> {"term":N,"candidate":"host:port","voter":"host:port"}
```

### Generation bump (the "log append" analog)

Every write through `_save_json_blob()` for a tracked path, **on the
writer only**, increments a single monotonic `generation` counter and
recomputes that path's SHA-256 in the manifest -- this is the direct
analog of picowal's `write_off` advancing on every WAL append.

### Replica poll loop (mirrors `picowal_repl_client_thread`)

A background asyncio task, only active while this node is a replica:

1. `GET {primary}/repl/status`. 3 consecutive failures marks the
   primary unhealthy (`REPL_HEALTH_FAIL_THRESHOLD`, same constant name
   and value as picowal's), driving gossip below -- **identical
   failure-detection design** to `picowal_repl_client_primary_healthy()`.
2. If `generation` advanced, diff the blob-hash manifest against the
   locally-cached one; for every path whose hash changed, `GET
   /repl/blob?path=...` and write it into **this pod's own local
   store** (no shared filesystem needed -- exactly picowal's value
   proposition, "replicate the log over HTTP instead of requiring
   shared storage").
3. Once all changed blobs are pulled, call the **same
   `_reload_all_persisted_state()`** function `app.py`'s own startup
   handler already uses to rebuild every in-memory structure from
   blobs (BM25 postings, vector matrix, promotions index, autocomplete
   vocabulary, rules, redirects, ranking objective, ML ranker
   snapshot) -- this is the direct analog of
   `picowal_db_repl_ingest()`.

### Mutation gating (mirrors replicas refusing local `/wal/` writes)

A `require_writer` FastAPI dependency, added to every endpoint that
mutates a tracked blob (`/search/ingest/catalog`,
`/search/ingest/from-erp`, `/search/ingest/promotions`,
`/search/admin/rules` POST/DELETE, `/search/admin/redirects`
POST/DELETE, `/search/admin/ranking-objective`,
`/search/admin/ai-toggle`, `/search/admin/vector-index/rebuild`,
`/search/admin/ml-model/reset`) -- returns `503 not_writer` on a
replica, exactly mirroring `picowal_replica_mode_enabled()` gating
`/wal/` mutation routes. `/search/events` (click/view/purchase
telemetry) is deliberately NOT gated -- see "what gets replicated"
above.

### Gossip leader election (mirrors `picowal_gossip`)

Identical design to `picowal_gossip.c`: a fixed, statically-configured
set of node ids (`WAVESEARCH_FOLLOWERS`, comma-separated
`host:port` -- populated from a Kubernetes **StatefulSet**'s stable
pod ordinals, e.g. `wavesearch-api-0.wavesearch-api-headless:8803,...`,
since gossip's deterministic-candidate trick requires stable identity,
which a plain `Deployment`'s randomly-named pods don't provide). Every
node ticks every `GOSSIP_TICK_S`; while its believed primary is
healthy, nothing happens; once unhealthy, it deterministically
nominates the lexicographically-smallest registered follower id as
candidate and gossips a vote via `POST /gossip/vote` to every peer.
The moment any node observes strictly-more-than-half the followers'
votes for a candidate, **every node** (not just the winner) records
that candidate as `known_leader`; the winning candidate additionally
self-promotes (stops its repl-client loop, starts serving its own
`/repl/status`+`/repl/blob`, flips `require_writer` to pass).

### Documented improvement over the picowal reference

`picowal_gossip.c`'s own doc comment acknowledges it's "deliberately
NOT a full consensus protocol" and only describes the *winning*
follower's self-promotion -- it doesn't specify how the *other*,
losing followers learn to re-point their `repl_client` at the new
leader (their `--picowal-replica-of` is static CLI config). This
implementation closes that gap cheaply since gossip vote-tallying is
symmetric (any node can independently compute "candidate X reached
quorum," not just X itself): **every** node updates `known_leader`
the moment it observes quorum for any candidate, and the replica poll
loop always targets `known_leader` if set, falling back to the
original static "assumed primary" (the deterministic candidate) only
before any election has ever completed. This is a genuine, explicit
enhancement over the reference design, not a silent behavior change --
call it out if porting this pattern back to picowal itself.

## What's NOT solved (explicit scope boundaries, same honesty standard
as every other `pico*` library's README in this session)

- **Still not a full consensus protocol.** No fencing tokens, no
  protection against a stale writer reappearing mid-election and both
  nodes briefly believing they're the writer (the same split-brain
  caveat `picowal_gossip.h` documents). Acceptable for a demo/dev
  platform; a production deployment fronting real revenue would want
  a real consensus system (etcd/Raft-backed leader election) instead.
- **Full-blob-manifest polling, not incremental deltas.** Every
  replica re-downloads and reloads the *entire* changed blob (not a
  diff) and calls a full in-memory rebuild -- fine at this catalog
  scale (matches the existing `_rebuild_runtime_from_snapshot`'s own
  "rebuild from the whole snapshot" pattern), not designed for
  megabyte-scale catalogs polled every 500ms.
- **Bootstrap ordering**: the very first StatefulSet rollout has no
  existing writer; every pod starts as a replica assuming the
  deterministic candidate is the (not-yet-running) primary, fails its
  first few polls, and elects that same candidate within one gossip
  tick -- a few seconds of "no writer available" (mutation endpoints
  503) on first cluster bring-up, not on ordinary rolling restarts of
  an already-converged cluster (the new pod just becomes a replica of
  the existing writer immediately).

## Files

| File | Purpose |
|---|---|
| `src/wavesearch_api/wavesearch_repl.py` | The whole replication engine: `ReplConfig`/`ReplState`, FastAPI router (`/repl/*`, `/gossip/*`), `require_writer` dependency, background poll/gossip asyncio tasks |
| `src/wavesearch_api/app.py` | Wires it in: tracked blob list, `_save_json_blob` generation-bump hook, `_reload_all_persisted_state()` (refactored out of the startup handler so the repl poller can call it too), `require_writer` added to every mutating endpoint |
| `k8s/base/wavesearch-api-statefulset.yaml` | Replaces the `Deployment` with a `StatefulSet` + headless `Service` (stable pod identity, required for the static follower list) |
| `scripts/wavesearch_repl_smoke_test.py` | Manual smoke test: starts 2 local uvicorn instances, ingests a catalog against the elected writer, confirms the replica picks it up, kills the writer, confirms the replica self-promotes and starts accepting writes |
